# Free API Switch 需求规格

## 1. 产品定位

Free API Switch 是一个运行在本机的轻量级大模型 API 代理服务。它通过工程目录下的配置文件管理多个上游大模型 API，并向 TRAE 或其他编程工具暴露一个固定的本机访问地址。

第一版重点不是协议转换器，也不是完整的模型网关，而是一个"本地固定入口 + 统一 local_key 鉴权 + 按端点家族路由到默认 profile + 用 `model_id` 覆盖上游模型名 + 原样转发 + 缺省参数补全"的代理工具。

## 2. 第一版目标

- 在本机启动一个 HTTP 代理服务，监听地址由 `app.json` 中的 `host` 配置，默认 `127.0.0.1`（仅本机）；也允许改为 `0.0.0.0` 或局域网 IP 以支持同网段设备接入，此时必须搭配足够强的 `local_key`。
- 编程工具通过固定本机地址访问该服务。
- 编程工具的 API Key 字段填写统一的本地 token（`local_key`），而不填写真实上游 API Key。
- 服务通过 `app.json` 中配置的单一 `local_key` 做鉴权，并按客户端请求的路径自动选择：
  - `/v1/chat/completions` → `app.default_openai_model`
  - `/v1/messages` → `app.default_anthropic_model`
  再在 `model_api.json` 中按 `model_name` 匹配对应 profile，并把请求转发到该 profile 的 `base_url`，同时用 profile 的 `model_id` 覆盖请求 body 的 `model` 字段。
- 支持 OpenAI 兼容接口和 Anthropic/Claude 兼容接口。
- 不做 OpenAI 与 Claude 之间的请求或响应格式转换。
- 支持流式响应透传。
- 支持对生成类端点进行缺省参数补全。
- 提供 CLI 命令：`fas start`、`fas check`、`fas list`。

## 2.1 下一迭代目标：Windows 托盘版

在第一版 CLI 代理服务稳定后，下一轮迭代增加 Windows 托盘运行形态：

- 提供可双击运行的 Windows 单文件便携 exe 交付物；exe 与 `cfgs/` 目录同级放置，即可开箱使用。
- 运行后默认不显示命令行窗口，以右下角任务栏托盘图标常驻。
- 托盘程序内置并管理 fas 本机 HTTP 服务，启动托盘即启动代理服务。
- 右键托盘菜单按 OpenAI / Anthropic 两个协议家族分组，分别列出 `model_api.json` 中所有 `enabled: true` 的 profile（radio 单选），当前默认模型打勾；单击任一项后：
  1. 立即将新模型名写回 `cfgs/app.json` 的 `default_openai_model` 或 `default_anthropic_model`；
  2. 热更新 `runtime.app` 内存字段，下一次请求即生效，无需重启进程；
  3. 通过系统 `Notification` 反馈 "已切换 xxx 默认模型：yyy"；
  4. 刷新菜单勾选状态。
- OpenAI 分组只列 `type: 'openai_compatible'` 的 profile；Anthropic 分组只列 `type: 'anthropic_compatible'` 的 profile；若某分组为空，显示灰色的 "(无 enabled … profile)"。
- 菜单末尾保留“退出”项，点击时优雅关闭本机 HTTP 服务并退出托盘程序。
- 托盘版必须限制单实例运行，避免重复启动造成端口占用或多个托盘图标。
- 启动失败时通过 Windows 系统通知反馈原因，并保留托盘图标以便用户退出。
- 托盘版配置查找规则：开发态（`npm run tray:dev`）沿用 CLI 的 `process.cwd()/cfgs`；打包 exe（`app.isPackaged === true`）从 exe 同级目录的 `cfgs/` 读取配置、向 exe 同级目录的 `logs/` 写日志。
- 保留调试模式入口，用于显示调试窗口或控制台信息。

## 3. 非目标

- 第一版不做 GUI。
- 第一版不做 Web 管理界面。
- 第一版 CLI 范围不做托盘应用；托盘应用作为下一迭代目标单独设计和实现。
- 第一版不提供公网固定 IP。
- 第一版不做模型名路由。
- 第一版不做端口路由。
- 第一版不做路径中的 profile 名路由。
- 第一版不做 OpenAI 与 Claude 的协议互转。
- 第一版不自造 `/v1/models` 模型列表。
- 第一版不统计 token 用量或费用。
- 第一版不支持配置热重载。

## 4. 运行方式

### 4.1 部署形态

第一版作为本机服务运行，主要面向 Windows，同时支持 Ubuntu。

默认监听：

```text
127.0.0.1:8787
```

监听地址由 `app.json` 的 `host` 字段决定，默认 `127.0.0.1`（仅本机可访问）。若改为 `0.0.0.0` 或局域网 IP（如 `192.168.1.100`），同网段的其他设备也能调用本代理，此时必须搭配足够强的 `local_key`；本项目不提供公网固定 IP。

### 4.2 技术栈

第一版使用 Node.js + TypeScript 实现。

具体 HTTP 框架暂不锁定，但实现必须满足：

- 支持 HTTP 请求代理。
- 支持流式响应透传。
- 支持 JSON 配置读取和严格校验。
- 支持跨平台路径处理。

### 4.3 托盘版运行方式

托盘版面向 Windows 优先交付，后续可保留跨平台扩展空间。

托盘版运行规则：

- 默认静默启动，不显示命令行窗口。
- 内置启动本机 HTTP 服务，服务行为与 `fas start` 保持一致。
- 配置文件读取、校验、日志目录和端口监听规则沿用 CLI 版本；打包 exe 运行时改从 exe 同级 `cfgs/`、`logs/` 读写（详见 2.1）。
- 只允许单实例运行；重复启动不得创建第二个代理服务。
- 右键菜单按 OpenAI / Anthropic 协议家族分组列出所有 `enabled: true` profile，以 radio 单选形式切换 `default_openai_model` / `default_anthropic_model`；当前默认模型打勾。切换后立即将新值写回 `cfgs/app.json`（`JSON.stringify` 缩进 2）、热更新 `runtime.app` 内存字段（下一次请求即生效，无需重启），并通过系统 `Notification` 反馈切换结果。菜单末尾始终保留“退出”项。
- 启动失败通过系统通知提示，不静默失败。
- 调试模式可显示窗口或控制台，便于排查配置错误、端口占用和启动失败。

## 5. 核心概念

### 5.1 Profile

一个 profile 表示一个"虚拟模型入口"。

一个 profile 绑定以下信息：

- 本地名称（`model_name`），用于 `app.default_openai_model` / `app.default_anthropic_model` 匹配。
- 上游模型 ID（`model_id`），真正下发到上游 API 的 `model` 字段值。
- 接口类型。
- 上游 API 根地址。
- 真实上游 API Key。
- 最大输出 token。
- 最大上下文 token。
- 默认 token 参数（兜底）。
- 是否启用。

`model_name` 与 `model_id` 分离是为了让本地"友好名字"与上游供应商"真实模型 ID"解耦：切换品牌/版本只需改 `model_id` 或 `app.default_*_model` 指向另一个 profile，无需变动其他配置。

编程工具不直接知道真实上游 API Key，只通过统一的本地 token 使用服务。

### 5.2 Local Key

`local_key` 是工具侧填写的 API Key，在 `app.json` 中配置，为全局统一的本地鉴权令牌。

本地服务收到请求后，从 `Authorization: Bearer <token>` 或 `x-api-key: <token>` 中解析 token，与 `app.json` 的 `local_key` 比对。匹配成功后再进入 profile 选择流程。

`local_key` 只用于本地鉴权，不会转发给上游供应商。

### 5.3 Default Model 与端点路由

`app.json` 中包含两个默认模型字段，各自服务于一个协议家族：

| 客户端请求路径              | 使用的默认字段              | 要求 profile.type       |
| --------------------------- | --------------------------- | ----------------------- |
| `POST /v1/chat/completions` | `default_openai_model`      | `openai_compatible`     |
| `POST /v1/messages`         | `default_anthropic_model`   | `anthropic_compatible`  |

命中默认字段后的行为：

1. **选择 profile**：在 `model_api.json` 中按 `model_name` 匹配对应的 profile。
2. **覆盖模型名**：将请求 body 中的 `model` 字段始终替换为该 profile 的 `model_id`。

启动校验时强制两个默认字段都必须指向对应类型的启用 profile，否则拒绝启动。

### 5.4 接口类型

第一版只支持两种接口类型：

- `openai_compatible`
- `anthropic_compatible`

接口类型用于决定上游鉴权 Header 的生成方式，同时约束这个 profile 能被哪个默认字段引用。

## 6. 配置文件

### 6.1 配置目录

第一版配置文件放在工程目录的 `cfgs` 文件夹下。

配置拆分为两个文件：

- `cfgs/app.json`：软件运行配置（含 `local_key`、`default_openai_model`、`default_anthropic_model`）。
- `cfgs/model_api.json`：上游 profile 列表。

真实配置文件会明文保存 API Key，因此不应提交到版本控制。仓库可以提供示例文件：

- `cfgs/app.example.json`
- `cfgs/model_api.example.json`

### 6.2 app.json 示例

```json
{
  "host": "127.0.0.1",
  "port": 8787,
  "log_level": "debug",
  "request_timeout_ms": 600000,
  "local_key": "local-your-token",
  "default_openai_model": "deepseek-chat",
  "default_anthropic_model": "claude-sonnet-4"
}
```

字段说明：

- `host`：监听地址，默认 `127.0.0.1`；允许任意非空字符串，例如 `localhost` / `0.0.0.0` / 具体局域网 IP。改为 `0.0.0.0` 或局域网 IP 会让同网段设备也能访问本代理，必须搭配足够强的 `local_key`。
- `port`：监听端口。
- `log_level`：日志级别，第一版默认 `debug`。
- `request_timeout_ms`：上游请求超时时间，采用长超时策略。
- `local_key`：全局统一的本地鉴权令牌，编程工具 API Key 字段填写此值。
- `default_openai_model`：`/v1/chat/completions` 请求默认使用的 profile 的 `model_name`；对应 profile 的 `type` 必须为 `openai_compatible`。
- `default_anthropic_model`：`/v1/messages` 请求默认使用的 profile 的 `model_name`；对应 profile 的 `type` 必须为 `anthropic_compatible`。

### 6.3 model_api.json 示例

```json
{
  "profiles": [
    {
      "model_name": "deepseek-chat",
      "model_id": "deepseek-chat",
      "enabled": true,
      "type": "openai_compatible",
      "base_url": "https://api.deepseek.com",
      "api_key": "real-upstream-api-key",
      "max_output_tokens": 8192,
      "max_context_tokens": 65536,
      "thinking_level": "medium",
      "defaults": {}
    },
    {
      "model_name": "claude-sonnet-4",
      "model_id": "claude-sonnet-4-20250514",
      "enabled": true,
      "type": "anthropic_compatible",
      "base_url": "https://api.anthropic.com",
      "api_key": "real-upstream-api-key",
      "max_output_tokens": 4096,
      "max_context_tokens": 200000,
      "defaults": {
        "anthropic_version": "2023-06-01"
      }
    }
  ]
}
```

字段说明：

- `model_name`：本地标识，用于 `app.default_openai_model` / `app.default_anthropic_model` 匹配；全局唯一。
- `model_id`：真正下发到上游 API 的模型名。
- `enabled`：是否启用，默认 `true`。
- `type`：接口类型，只能是 `openai_compatible` 或 `anthropic_compatible`。
- `base_url`：上游 API 根地址，不包含 `/v1`。
- `api_key`：真实上游 API Key，明文存储在本地配置文件中。
- `max_output_tokens`：最大输出 token 数（可选）。请求缺少 `max_tokens` 时优先使用此值。
- `max_context_tokens`：最大上下文长度 token 数（可选）。请求缺少 `max_context_tokens` 时优先使用此值。
- `thinking_level`：思考层级（可选，`string | number`）。请求缺少 `thinking_level` 时优先使用此值；已有值时不动。
- `defaults`：生成类端点的默认参数兜底（可选）。只在请求缺少对应字段时补全，优先级低于 `max_output_tokens`、`max_context_tokens` 和 `thinking_level`。

## 7. 路由与转发规则

### 7.1 Profile 选择

第一版使用 `app.json` 中统一的 `local_key` 鉴权，再按客户端请求路径匹配对应 profile。

流程：

1. 工具侧请求携带 `Authorization: Bearer <local_key>` 或 `x-api-key: <local_key>`。
2. 服务解析 token，与 `app.json` 的 `local_key` 比对。
3. 依据请求路径判定协议家族：
   - `/v1/chat/completions` → openai
   - `/v1/messages` → anthropic
   - 其他生成类端点 → 直接返回 404 `UNSUPPORTED_ENDPOINT`
4. 用对应家族的 `app.default_*_model` 在 `model_api.json` 的 profiles 列表中按 `model_name` 字段查找匹配项。
5. 找到对应的 profile 后，进入参数补全与转发流程。

典型工具配置：

```text
Base URL: http://127.0.0.1:8787/v1
API Key: local-your-token
Model:   任意（会被 profile.model_id 覆盖）
```

### 7.2 路径拼接

`base_url` 不包含 `/v1`。

本地服务收到请求后，将请求路径原样追加到 profile 的 `base_url` 后面。

示例：

```text
本地请求: http://127.0.0.1:8787/v1/chat/completions
profile.base_url: https://api.deepseek.com
上游请求: https://api.deepseek.com/v1/chat/completions
```

### 7.3 鉴权 Header

本地服务根据 profile 的 `type` 自动生成上游鉴权 Header。

`openai_compatible`：

```text
Authorization: Bearer {api_key}
```

`anthropic_compatible`：

```text
x-api-key: {api_key}
```

工具传来的本地鉴权信息不会转发给上游。

## 8. 参数补全规则

第一版只对生成类端点进行缺省参数补全：

- `/v1/chat/completions`
- `/v1/messages`

补全原则：

- `model`：始终使用当前 profile 的 `model_id` 覆盖请求中的值。
- `max_tokens`：请求缺失时，优先用 `max_output_tokens`，其次用 `defaults.max_tokens`；已有值时不动。
- `max_context_tokens`：请求缺失时，优先用 `max_context_tokens`，其次用 `defaults.max_context_tokens`；已有值时不动。
- `thinking_level`：请求缺失时，优先用 profile 的 `thinking_level`，其次用 `defaults.thinking_level`；已有值时不动。
- 其余字段：请求缺失时，从 `defaults` 补全（跳过已由专用字段处理的 `max_tokens`、`max_context_tokens` 和 `thinking_level`）。
- 不覆盖工具已经传入的字段（`model` 除外，始终覆盖）。
- 不做 OpenAI 与 Claude 的字段互转。
- 非生成类端点原样转发。

示例：

- 请求的 `model` 始终被替换为当前 profile 的 `model_id`。
- 请求缺少 `max_tokens` 时，补 `max_output_tokens`。
- 请求缺少 `max_context_tokens` 时，补 `max_context_tokens`。
- 请求缺少 `thinking_level` 时，补 profile 的 `thinking_level`。
- 请求已经包含 `max_tokens` 时，不覆盖。

## 9. 流式响应

第一版必须支持 `stream=true` 的流式响应透传。

流式响应处理原则：

- 不解析流内容。
- 不转换流格式。
- 不缓存完整响应。
- 尽量按上游响应原样转发给客户端。

## 10. 错误处理

### 10.1 上游错误

上游供应商返回错误时，本地服务原样返回：

- 保留上游 HTTP 状态码。
- 保留上游响应体。
- 不包装成统一错误格式。

### 10.2 本地错误

本地服务自身错误使用本地错误响应，例如：

- 配置文件不存在。
- 配置校验失败。
- `local_key` 无效。
- profile 被禁用。
- 上游连接失败。
- 生成类端点需要补参但请求体不是合法 JSON。

## 11. 日志

第一版默认使用 `debug` 日志级别。

日志分为两类，采用不同的落盘策略：

**运行日志（debug / info / warn / error / access）**

- 同时输出到控制台和文件。
- 文件写入 `logs/` 目录，按天轮转，文件名格式为 `fas-YYYY-MM-DD.log`。
- 采用 JSON-line 格式（每行一个 JSON 对象）。

**消息日志（转发出去的请求 messages 与大模型返回的响应内容）**

- 仅在命中生成类端点（如 `/v1/chat/completions`、`/v1/messages`）时产生。
- 每次调用单独写入一个 JSON 文件，路径为 `logs/messages/<ISO 时间戳>.json`。
- 文件内容为结构化 JSON，字段包含 `ts`、`request_messages`、`response_message`。
- `request_messages` 是本次实际发送给上游 API 的**完整 messages 数组**，包含 system prompt、上层软件插入的提示词、历史对话与本次用户输入；`response_message` 是本次上游返回的助手消息（单条）。
- 时间戳作为文件名时冒号和点会被替换为 `-`，避免文件系统冲突。

日志需要支持详细调试，但必须遵守脱敏规则。

运行日志允许记录：

- 请求时间。
- profile 名称。
- profile 类型。
- 请求路径。
- 上游 base_url。
- HTTP 状态码。
- 请求耗时。
- 是否命中默认参数补全。

消息日志允许记录（仅落盘到 `logs/messages/`，不进入运行日志）：

- `request_messages`：本次实际发送给上游 API 的完整 messages 数组（包含 system prompt、软件插入的提示词、历史对话与本次用户输入，按上游协议原样保留顺序）。
- `response_message`：本次上游返回的助手消息。非流式响应直接取（OpenAI 兼容取 `choices[0].message`；Anthropic 兼容取 `{ role, content }`）；SSE 流式响应聚合 delta 事件后组装为单个 message。

禁止记录：

- 真实 `api_key`。
- `local_key` 明文。
- `Authorization` 明文。
- `x-api-key` 明文。
- Cookie 明文。
- 未经脱敏的完整请求正文。
- 未经脱敏的完整响应正文。
- 用户 prompt、代码片段或模型输出全文（脱敏后可记录摘要）。

## 12. CLI 命令

CLI 命令名为 `fas`。

### 12.1 fas start

启动本机代理服务。

启动流程：

1. 读取 `cfgs/app.json`。
2. 读取 `cfgs/model_api.json`。
3. 严格校验配置。
4. 监听 `app.host:app.port`（默认 `127.0.0.1:8787`）。
5. 开始转发请求。

### 12.2 fas check

校验配置文件。

校验内容包括：

- `app.json` 是否存在。
- `model_api.json` 是否存在。
- JSON 格式是否合法。
- `host` 是否为非空字符串（不再强制等于 `127.0.0.1`）。
- `port` 是否合法。
- `local_key` 是否存在且非空。
- `default_openai_model` 是否存在且非空。
- `default_anthropic_model` 是否存在且非空。
- profile 的 `model_name` 是否全局唯一。
- 每个启用 profile 的 `model_id` 是否存在且非空。
- 启用 profile 的 `type` 是否合法。
- 启用 profile 的 `base_url` 是否不包含 `/v1`。
- 启用 profile 的 `api_key` 是否存在。
- `default_openai_model` 是否匹配一个启用的、`type=openai_compatible` 的 profile。
- `default_anthropic_model` 是否匹配一个启用的、`type=anthropic_compatible` 的 profile。

### 12.3 fas list

列出 profiles。

输出内容包括：

- `model_name`
- `model_id`
- `enabled`
- `type`
- `base_url`

`fas check` 会额外打印 `default_openai_model` 与 `default_anthropic_model`。

输出内容不得显示真实 `api_key` 或 `local_key` 明文。

## 13. 配置变更

第一版配置在服务启动时读取一次。

修改 `app.json` 或 `model_api.json` 后，需要重启服务才会生效。

切换某个协议下的默认模型的方式：修改 `app.json` 对应的 `default_openai_model` 或 `default_anthropic_model`（指向 `model_api.json` 中另一个启用且类型匹配的 profile 的 `model_name`），然后重启服务。

## 14. 超时与重试

第一版采用长超时、不自动重试策略。

- 超时时间由 `app.json` 的 `request_timeout_ms` 控制。
- 生成类请求失败后不自动重试。
- 避免重复扣费和重复生成。

## 15. 配置校验策略

第一版采用严格校验。

只要启用的 profile 存在配置错误，服务拒绝启动。

`enabled: false` 的 profile 不参与 `default_openai_model` / `default_anthropic_model` 匹配，也不阻塞服务启动。
