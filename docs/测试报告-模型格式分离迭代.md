# Free API Switch — 模型格式分离迭代测试报告

- 报告日期：2026-07-04
- 迭代目标：
  - 按客户端请求路径的“协议家族”选择默认 profile：`/v1/chat/completions` → `default_openai_model`；`/v1/messages` → `default_anthropic_model`；其他生成类端点直接返回 404 `UNSUPPORTED_ENDPOINT`。
  - profile 拆分为 `model_name`（本地标识，用于与两个默认字段匹配）与 `model_id`（真正下发到上游 API 的模型名）。
  - `app.json` 移除 `default_model`，新增 `default_openai_model` 与 `default_anthropic_model`。
  - 请求补全时 `body.model` 覆盖为 `profile.model_id`（不再是 `app.default_model`）。
- 测试范围：配置 schema/校验、profile 解析、参数补全、请求处理流水线、CLI（check/list）、端到端真实上游调用。

## 2. 被测代码变更清单

核心源文件：

- `src/core/types.ts`：`AppConfig` 新增 `default_openai_model` / `default_anthropic_model`；`ProfileConfig` 拆分 `model_name` + `model_id`；新增 `EndpointFamily` 类型；`RuntimeConfig` 用 `enabledProfilesByModelName` 索引替代原 `enabledProfilesByName`。
- `src/core/profileResolver.ts`：新增 `resolveEndpointFamily(pathname)`；`resolveProfile(runtime, localKey, family)` 增加 `family` 参数，按家族选择默认字段并从 `enabledProfilesByModelName` 查找 profile。
- `src/core/defaults.ts`：`applyDefaults(bodyText, profile)` 不再需要 `defaultModel` 参数；`body.model` 始终覆盖为 `profile.model_id`。
- `src/config/schema.ts`：`appSchema` 新增两个默认字段；`profileSchema` 用 `model_name` + `model_id` 替换 `name`。
- `src/config/validateConfig.ts`：新增两个默认字段与对应 profile 类型的交叉校验（`default_openai_model` 必须匹配 `openai_compatible` 类型的启用 profile；`default_anthropic_model` 同理）；构建 `enabledProfilesByModelName`。
- `src/server/requestHandler.ts`：主流程改为先 `resolveEndpointFamily`，未匹配到家族返回 404 `UNSUPPORTED_ENDPOINT`，随后 `resolveProfile(runtime, localKey, family)`。
- `src/server/app.ts`：启动信息打印两个默认字段。
- `src/cli/commands.ts`：`fas check` 打印两个默认；`fas list` 表格列改为 `model_name / model_id / enabled / type / base_url`。
- `cfgs/*.example.json`：示例配置更新为新格式（两个默认字段 + profile 使用 `model_name` + `model_id`）。

## 6. 兼容性

- 本轮属于**破坏性变更**，无向后兼容。
- 旧格式 `cfgs/app.json`（含 `default_model`）与 `cfgs/model_api.json`（profile 用 `name`）不再被 schema 接受；`fas check` 会直接返回 `CONFIG_VALIDATION_FAILED`。
- 升级步骤（需用户手动完成）：
  1. `app.json`：删除 `default_model`；新增 `default_openai_model`（值为原来指向的 openai_compatible profile 标识）与 `default_anthropic_model`（值为对应 anthropic_compatible profile 标识）。
  2. `model_api.json`：将每个 profile 的 `name` 字段改为 `model_name`；新增 `model_id` 字段（真正下发给上游 API 的模型名）。若原来两个字段等价，可复用原值。
  3. 执行 `fas check` 验证升级结果。
