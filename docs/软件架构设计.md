# Free API Switch 软件架构设计

## 1. 设计依据

本文基于 `docs/需求规格.md` 进行第一版软件架构设计，目标是为后续软件详细设计和 spec 提供稳定的架构边界。

第一版定位为本机轻量级大模型 API 代理服务，核心能力是"固定本机入口 + 统一 local_key 鉴权 + 按请求路径的协议家族选择默认 profile 并用 profile.model_id 覆盖模型名 + 原样转发 + 缺省参数补全"。第一版不做协议互转、不做完整模型网关、不做 GUI、不做热重载、不做 token 统计。下一迭代在不破坏 CLI 入口的前提下增加 Windows 托盘运行形态。

## 2. 架构目标

### 2.1 功能目标

- 在本机启动 HTTP 服务，绑定 `runtime.app.host:runtime.app.port`，默认 `127.0.0.1:8787`；`host` 允许任意非空字符串（`localhost` / `0.0.0.0` / 局域网 IP 等）。
- 通过工具侧 API Key 字段传入的 token 与 `app.json` 的 `local_key` 匹配做本地鉴权。
- 根据客户端请求路径的协议家族选择默认 profile：`/v1/chat/completions` → `default_openai_model`；`/v1/messages` → `default_anthropic_model`；其他生成类端点返回 404 `UNSUPPORTED_ENDPOINT`。
- 通过 `default_openai_model` / `default_anthropic_model` 在 `model_api.json` 中按 `model_name` 字段匹配启用 profile，并使用 profile 的 `model_id` 作为真正下发到上游 API 的模型名。
- 根据 profile 转发到对应上游 API 根地址。
- 支持 `openai_compatible` 与 `anthropic_compatible` 两类接口。
- 对 `/v1/chat/completions` 与 `/v1/messages` 执行缺省参数补全。
- 支持流式响应透传，不解析、不转换、不缓存完整流内容。
- 提供 `fas start`、`fas check`、`fas list` 三个 CLI 命令。
- 下一迭代提供 Windows 托盘入口，托盘程序内置并管理本机 HTTP 服务。
- 托盘右键菜单按 OpenAI / Anthropic 家族分组列出所有 enabled profile，支持切换默认模型（写回 `cfgs/app.json` 并热更新内存运行时），并保留"退出"项。
- 托盘右键菜单提供"配置管理..."入口，弹出本地设置窗口对 `cfgs/app.json` 与 `cfgs/model_api.json` 进行完整的可视化编辑（含 profile 增删改、启用/禁用、抽象推理强度等级选择、`defaults` JSON 编辑），保存时先做 schema + 业务级校验，通过后原子写盘，然后自动重启 HTTP runtime 让新配置立即生效。
- 托盘版支持单实例运行、系统通知失败反馈和调试模式入口。

### 2.2 质量目标

- 安全：真实上游 API Key、local_key、鉴权 Header、请求正文、响应正文不得进入日志。
- 简单：第一版避免引入模型路由、协议转换、配置热重载等复杂能力。
- 可测试：配置校验、profile 匹配、参数补全、Header 重写、端点家族路由应具备独立单元测试边界。
- 可扩展：未来可以在不破坏核心代理链路的前提下扩展新接口类型、新 CLI 命令或 Web UI。
- 跨平台：路径、进程启动、文件读取兼容 Windows 与 Ubuntu。
- 桌面化：托盘版优先支持 Windows，同时避免把核心服务逻辑绑定到桌面框架。

## 3. 总体架构

```text
┌──────────────────────┐
│ TRAE / 编程工具       │
│ Base URL + local_key │
└──────────┬───────────┘
           │ HTTP
           ▼
┌──────────────────────┐
│ Local HTTP Server     │
│ 127.0.0.1:8787        │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│ Request Pipeline      │
│ auth → family → profile│
│ → body defaults → proxy│
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│ Upstream API Provider │
│ OpenAI-compatible or  │
│ Anthropic-compatible  │
└──────────────────────┘
```

系统采用单进程 CLI + HTTP 服务架构。CLI 负责命令解析、配置加载、配置校验和服务启动；HTTP 服务负责接收本机请求，并通过请求处理流水线完成鉴权、端点家族解析、profile 匹配、参数补全、Header 重写和上游转发。

下一迭代增加桌面托盘入口后，系统形成双入口架构：CLI 入口继续提供 `fas start/check/list`；托盘入口负责桌面生命周期、托盘菜单、单实例锁和系统通知。两类入口必须复用同一套配置加载、日志初始化和 HTTP 服务启动/关闭能力，不复制代理流水线。

## 4. 推荐技术方案

### 4.1 运行时与语言

- Node.js + TypeScript。
- 运行目标建议选择当前 LTS 版本 Node.js。
- 使用 ESM 或 CommonJS 需在详细设计阶段统一；第一版建议优先选择 ESM，便于后续与现代工具链集成。

### 4.2 HTTP 框架建议

第一版 HTTP 框架建议优先考虑 Node.js 原生 `http` 模块或轻量框架。

推荐方案：原生 `http` 模块。

原因：

- 需求主要是代理转发，不需要复杂路由系统。
- 原生请求与响应流更容易保持透传语义。
- 依赖更少，降低第一版实现和排障复杂度。
- 对流式响应、超时、AbortController 等控制更直接。

如果后续详细设计发现需要更强路由、中间件或插件生态，可以再评估 Fastify 等框架，但第一版不建议为了简单 CLI 代理引入过重 Web 框架。

### 4.3 上游请求实现建议

推荐使用 Node.js 内置 `fetch` 与 Web Streams，或在需要更细粒度 Node Stream 控制时使用 `http`/`https` 原生客户端。

详细设计阶段需要确认目标 Node.js 版本。如果目标版本稳定支持内置 `fetch`，则优先使用内置能力，减少依赖。如果遇到流式透传、Header 过滤或超时控制限制，再回退到原生 `http`/`https` 客户端。

### 4.4 配置校验建议

推荐使用 schema 校验库进行严格校验，例如 `zod` 或同类库。

校验是第一版核心边界之一，不建议手写零散校验逻辑。schema 应同时服务于：

- `fas start` 启动前校验。
- `fas check` 独立校验。
- 单元测试中的错误用例覆盖。
- 后续 spec 中的数据结构定义。

### 4.5 桌面托盘技术方案

下一迭代托盘版推荐使用 Electron 主进程实现。

原因：

- Electron 原生支持系统托盘、右键菜单、单实例锁和系统通知。
- 可以在主进程中直接复用现有 Node.js 服务模块，不需要额外启动 `fas start` 子进程。
- 支持 Windows 单文件便携 exe 打包（`electron-builder --win portable`），符合当前交付目标。
- 后续若需要 macOS 或 Linux 托盘形态，Electron 仍有扩展空间。

托盘模块除维护托盘生命周期外，还承担一项运行期交互职责：**切换默认模型**。托盘右键菜单按 OpenAI / Anthropic 家族分组列出所有 enabled profile，用户选择新的默认模型后，托盘负责将新值写回 `cfgs/app.json`（`JSON.stringify` 缩进 2）并热更新 `runtime.app` 内存字段。因此托盘模块需要引用共享运行时中的 `cfgDir` 字段定位 `app.json`。

托盘版不引入渲染层 UI 作为第一版目标。默认运行时不显示窗口，仅创建托盘图标和右键菜单；调试模式可打开窗口或控制台用于排障。

## 6. 模块设计

### 6.1 CLI 模块

职责：

- 解析 `fas start`、`fas check`、`fas list`。
- 定位工程目录下的 `cfgs/app.json` 与 `cfgs/model_api.json`。
- 调用配置加载与校验模块。
- 在 `fas start` 中启动 HTTP 服务。
- 在 `fas list` 中输出脱敏 profile 列表。

CLI 模块不应包含代理转发细节。这样可以保证 `fas check` 和 `fas list` 不需要启动服务，也不会误触发网络请求。

下一迭代需要将 `fas start` 中的启动逻辑整理为可复用函数，供 CLI 与托盘入口共同调用。

### 6.1.1 桌面托盘模块

职责：

- 创建 Windows 系统托盘图标。
- 创建右键菜单：按 OpenAI / Anthropic 家族分组列出 `enabled` profile，radio 单选切换 `default_openai_model` / `default_anthropic_model`；某分组无匹配 profile 时显示灰色 `(无 enabled xxx profile)`；菜单末尾保留“退出”项。
- 用户点击某个模型项时，读取 `cfgs/app.json`、修改对应默认字段、以缩进 2 的 JSON 写回，同时热更新 `running.runtime.app` 对应字段（下一次请求即生效），并通过系统 `Notification` 反馈切换结果；写入失败时通知报错并保留旧的内存值。
- 每次切换后重建菜单以刷新勾选状态。
- 提供"配置管理..."菜单项：打开一个本地设置窗口（Electron `BrowserWindow` + 内置 HTML/JS，不引入前端框架），加载当前 `app.json` 与 `model_api.json` 供用户编辑；保存时通过 IPC 调回主进程，复用 `validateConfig` 做校验，通过后以 `.tmp → rename` 原子写盘，再调用 `restartRuntime()` 关闭旧 server 并按同一 `cfgDir`/`logDir` 重新启动 runtime，让新配置立即生效。校验失败不写盘、不重启，UI 显示错误。
- 获取单实例锁，避免重复启动代理服务。
- 调用共享启动逻辑加载配置、创建 logger 并启动 HTTP 服务。
- 打包 exe 运行时（`app.isPackaged === true`），显式向共享启动函数传入 `cfgDir = <exe 同级>/cfgs`、`logDir = <exe 同级>/logs`；开发态沿用默认 `process.cwd()/cfgs`、`process.cwd()/logs`。
- 启动失败时发送系统通知，保留托盘图标以便用户退出。
- 点击“退出”时优雅关闭 HTTP 服务并退出应用。
- 支持调试模式入口。

托盘模块需要引用共享运行时返回的 `cfgDir` 字段定位 `app.json`，进行读-改-写。托盘模块不得绕过配置校验、不得复制请求处理流水线，也不得通过子进程执行 `fas start` 来启动服务。

### 6.2 配置模块

职责：

- 读取 `cfgs/app.json`。
- 读取 `cfgs/model_api.json`。
- 执行 JSON 解析与 schema 校验。
- 执行业务级严格校验。
- 输出运行期不可变配置对象。

业务级校验包括：

- `host` 必须为非空字符串（默认 `127.0.0.1`；允许 `localhost` / `0.0.0.0` / 局域网 IP 等任意值）。
- `port` 必须为合法端口。
- `local_key` 必须存在且非空。
- `default_openai_model` 与 `default_anthropic_model` 必须存在且非空。
- profile `model_name` 必须唯一。
- 必须存在 `model_name` 等于 `default_openai_model` 且 `type` 为 `openai_compatible` 的启用 profile。
- 必须存在 `model_name` 等于 `default_anthropic_model` 且 `type` 为 `anthropic_compatible` 的启用 profile。
- 启用 profile 的 `type` 必须为允许值。
- 启用 profile 的 `base_url` 不得包含 `/v1`。
- 启用 profile 必须配置 `api_key` 与 `model_id`。
- 禁用 profile 不参与默认字段匹配，也不阻塞服务启动。

配置模块应返回结构化错误，供 CLI 输出人类可读信息。

### 6.3 Profile 解析模块

职责：

- 从请求鉴权信息中解析 token。
- 与 `app.json` 的 `local_key` 比对做鉴权。
- 根据请求路径解析端点家族（`resolveEndpointFamily(pathname)`）：`/v1/chat/completions` → `openai`；`/v1/messages` → `anthropic`；其他生成类端点返回 404 `UNSUPPORTED_ENDPOINT`。
- 鉴权通过且端点家族识别后，用对应家族的默认字段（`default_openai_model` 或 `default_anthropic_model`）在 `model_api.json` 的启用 profiles 中按 `model_name` 查找匹配项。
- 对 local_key 缺失、无效、profile 未找到等情况返回本地错误。

鉴权解析支持常见工具传参方式：

- `Authorization: Bearer <token>`。
- 详细设计阶段再确认是否支持 `x-api-key: <token>`。

第一版不通过模型名、端口或 query 参数选择 profile。始终由请求路径决定协议家族，并由该家族对应的默认字段决定使用哪个 profile。

### 6.4 参数补全模块

职责：

- 判断请求路径是否属于生成类端点（并确定其协议家族）。
- 对合法 JSON 请求体执行缺省参数补全。
- `model`：始终覆盖为当前 profile 的 `model_id`（真正下发到上游 API 的模型名）。
- `max_tokens`：缺失时优先用 `max_output_tokens`，其次用 `defaults.max_tokens`，已有值不覆盖。
- `max_context_tokens`：缺失时优先用 `max_context_tokens`，其次用 `defaults.max_context_tokens`，已有值不覆盖。
- 其余字段：缺失时从 `defaults` 补全。
- 非生成类端点不读取、不改写请求体，直接进入代理转发。

**抽象推理强度（thinking_level）**：本项目定义一组抽象等级 `off / low / medium / high / xhigh`，请求转发时根据 `profile.type` 自动翻译成上游真实字段（`openai_compatible → reasoning_effort`；`anthropic_compatible → thinking { type, budget_tokens }`）。等级到上游字段的映射由 `core/types.ts` 中的 `THINKING_LEVEL_MAP` 作为单一数据源，`core/defaults.ts`、`config/schema.ts`、托盘配置窗口 UI 下拉框全部从该 map 派生，不做重复硬编码。生效优先级：客户端 body 已带上游真实字段（`reasoning_effort` / `reasoning` / `thinking`）→ 完全透传；否则 `body.thinking_level > profile.thinking_level`；翻译后 `body.thinking_level` 这个自定义字段会从请求体中剥离，避免上游收到未知字段。

补全规则：

- `/v1/chat/completions`（openai 家族）：补 `model`（始终覆盖为 `profile.model_id`）、`max_tokens`、`max_context_tokens` 与 `defaults` 中存在且请求缺失的字段。
- `/v1/messages`（anthropic 家族）：同上。
- 不做 OpenAI 与 Claude 字段互转。
- 请求体不是合法 JSON 且需要补参时，返回本地错误。

### 6.5 Header 重写模块

职责：

- 移除工具侧本地鉴权信息。
- 根据 profile 类型生成上游鉴权 Header。
- 过滤不应透传或可能冲突的 Hop-by-hop Header。
- 保留业务相关 Header，避免破坏供应商兼容接口。

Header 规则：

- `openai_compatible`：生成 `Authorization: Bearer {api_key}`。
- `anthropic_compatible`：生成 `x-api-key: {api_key}`。
- local_key 不得转发到上游。
- 日志中不得记录鉴权 Header 明文。

### 6.6 代理转发模块

职责：

- 将本地请求路径原样追加到 `profile.base_url`。
- 保留 query string。
- 转发 HTTP method、Header 和 body。
- 执行上游请求超时控制。
- 将上游响应状态码、响应 Header 和响应体返回给客户端。
- 对流式响应进行透传，不缓存完整响应。

上游错误处理原则：

- 上游返回 HTTP 错误时，保留状态码与响应体。
- 不包装成统一错误格式。
- 上游连接失败、超时等本地代理错误，返回本地错误响应。

### 6.7 HTTP 服务模块

职责：

- 绑定 `runtime.app.host:runtime.app.port`（默认 `127.0.0.1:8787`，`host` 可为任意非空字符串）。
- 为每个请求构造 request context。
- 串联鉴权、profile 解析、参数补全、Header 重写和代理转发。
- 在请求结束后记录脱敏访问日志（同时输出到控制台和文件）。
- 在服务启动失败时向 CLI 返回明确错误。

请求处理流水线：

```text
receive request
  → parse token from Authorization
  → validate against app.local_key
  → resolveEndpointFamily(pathname): /v1/chat/completions→openai / /v1/messages→anthropic / 其他→404 UNSUPPORTED_ENDPOINT
  → resolve profile by family's default field (default_openai_model 或 default_anthropic_model) matching profile.model_name
  → build target url
  → maybe apply defaults (model 覆盖为 profile.model_id)
  → rewrite upstream headers
  → proxy request
  → stream upstream response
  → write sanitized log (console + file)
```

### 6.8 日志模块

职责：

- 根据 `app.log_level` 控制运行日志级别。
- 提供统一脱敏函数。
- 运行日志同时输出到控制台和 `logs/` 目录下的日志文件，文件按天轮转，文件名格式为 `fas-YYYY-MM-DD.log`，采用 JSON-line 格式（每行一个 JSON 对象）。
- 消息日志（转发出去的请求 messages 与大模型返回的响应内容）每次调用单独写入一个 JSON 文件，路径为 `logs/messages/<ISO 时间戳>.json`，与运行日志分离；请求侧记录发送给上游 API 的完整 messages 数组（含 system、软件插入的提示词、历史对话与本次用户输入），响应侧记录本次上游返回的助手消息（单条）。
- 输出允许记录的请求元数据。
- 阻止敏感信息通过错误对象、Header 或 body 泄露。

运行日志允许记录：

- 请求时间。
- profile 本地标识（model_name）。
- profile 类型。
- 端点家族（openai / anthropic）。
- 请求路径。
- 上游 base_url。
- HTTP 状态码。
- 请求耗时。
- 是否命中默认参数补全。

消息日志允许记录（仅落盘到 `logs/messages/`，不进入运行日志）：

- `request_messages`：本次实际发送给上游 API 的完整 messages 数组（包含 system prompt、软件插入的提示词、历史对话与本次用户输入，按上游协议原样保留顺序）。
- `response_message`：本次上游返回的助手消息。非流式响应直接取（OpenAI 兼容取 `choices[0].message`；Anthropic 兼容取 `{ role, content }`）；SSE 流式响应聚合 delta 事件后组装为单个 message。

禁止记录：

- 真实 `api_key`。
- `local_key` 明文。
- `Authorization` 明文。
- `x-api-key` 明文。
- Cookie 明文。
- 未经脱敏的完整请求正文。
- 未经脱敏的完整响应正文。
- 用户 prompt、代码片段或模型输出全文（脱敏后可记录摘要）。

## 7. 核心数据结构

### 7.1 AppConfig

```ts
type AppConfig = {
  host: string
  port: number
  log_level: 'debug' | 'info' | 'warn' | 'error'
  request_timeout_ms: number
  local_key: string
  default_openai_model: string
  default_anthropic_model: string
}
```

`host` 默认 `127.0.0.1`，允许任意非空字符串。

### 7.2 ProfileConfig

```ts
type ProfileConfig = {
  model_name: string      // 本地标识，用于与 default_openai_model / default_anthropic_model 匹配
  model_id: string        // 真正下发到上游 API 的模型名
  enabled: boolean
  type: 'openai_compatible' | 'anthropic_compatible'
  base_url: string
  api_key: string
  max_output_tokens?: number
  max_context_tokens?: number
  defaults: Record<string, unknown>
}
```

### 7.3 EndpointFamily

```ts
type EndpointFamily = 'openai' | 'anthropic'
```

### 7.4 RuntimeConfig

```ts
type RuntimeConfig = {
  app: AppConfig
  profiles: ProfileConfig[]
  /** 启用 profile 按 model_name 建立的索引，用于按端点家族路由查找 */
  enabledProfilesByModelName: Map<string, ProfileConfig>
}
```

运行期配置应在启动时一次性构建，服务运行期间不热更新。

## 8. 请求链路设计

### 8.1 OpenAI 兼容请求

```text
client
  POST http://127.0.0.1:8787/v1/chat/completions
  Authorization: Bearer local-deepseek-token
        │
        ▼
local server
  validate token against app.local_key ✓
  resolveEndpointFamily("/v1/chat/completions") → openai
  resolve profile by app.default_openai_model → "deepseek-v4-flash" → profile whose model_name = "deepseek-v4-flash"
  override body.model with profile.model_id
  fill missing max_tokens / max_context_tokens
  rewrite Authorization with upstream api_key
        │
        ▼
upstream
  POST https://api.deepseek.com/v1/chat/completions
  Authorization: Bearer <upstream-api-key>
```

### 8.2 Anthropic 兼容请求

```text
client
  POST http://127.0.0.1:8787/v1/messages
  Authorization: Bearer local-claude-token
        │
        ▼
local server
  validate token against app.local_key ✓
  resolveEndpointFamily("/v1/messages") → anthropic
  resolve profile by app.default_anthropic_model → "claude-sonnet-4-6" → profile whose model_name = "claude-sonnet-4-6"
  override body.model with profile.model_id
  fill missing max_tokens / max_context_tokens / anthropic_version
  remove local Authorization
  add x-api-key with upstream api_key
        │
        ▼
upstream
  POST https://api.anthropic.com/v1/messages
  x-api-key: <upstream-api-key>
```

### 8.3 未支持端点

```text
client
  POST http://127.0.0.1:8787/v1/unknown
        │
        ▼
local server
  resolveEndpointFamily("/v1/unknown") → 无匹配家族
  → 直接返回 HTTP 404 UNSUPPORTED_ENDPOINT，不落到上游
```

## 9. 错误处理架构

错误分为两类：上游错误与本地错误。

### 9.1 上游错误

上游已经返回 HTTP 响应时，本地服务尽量原样返回：

- 保留 HTTP 状态码。
- 保留响应体。
- 不改写为统一错误结构。
- 仅做必要 Header 过滤。

### 9.2 本地错误

本地服务自身错误使用统一的本地错误响应结构。建议结构如下：

```json
{
  "error": {
    "code": "INVALID_LOCAL_KEY",
    "message": "Invalid local API key"
  }
}
```

建议错误码：

- `CONFIG_FILE_NOT_FOUND`
- `CONFIG_JSON_INVALID`
- `CONFIG_VALIDATION_FAILED`
- `LOCAL_KEY_MISSING`
- `INVALID_LOCAL_KEY`
- `UNSUPPORTED_ENDPOINT`
- `PROFILE_NOT_FOUND`
- `PROFILE_DISABLED`
- `REQUEST_BODY_INVALID_JSON`
- `UPSTREAM_CONNECTION_FAILED`
- `UPSTREAM_TIMEOUT`
- `INTERNAL_ERROR`

错误响应不得包含真实 API Key、local_key 明文、请求正文或上游响应正文的敏感片段。

## 10. 配置与安全设计

### 10.1 配置文件边界

- 真实配置文件：`cfgs/app.json`、`cfgs/model_api.json`。
- 示例配置文件：`cfgs/app.example.json`、`cfgs/model_api.example.json`。
- 真实配置文件包含明文 API Key，不进入版本控制。
- 示例配置只放假数据，不放真实 token。

### 10.2 本地监听边界

- `app.host` 只做非空校验，默认 `127.0.0.1`；允许 `localhost` / `0.0.0.0` / 局域网 IP 等任意非空字符串。
- 一旦 `host` 改成 `0.0.0.0` 或具体局域网 IP，同网段任意设备只要拥有 `local_key` 就能访问真实上游 API；建议同步把 `local_key` 换成长随机字符串。
- 本项目不提供公网固定 IP，也不提供反向代理或 TLS 卸载。

### 10.3 Secret 处理边界

- `api_key` 只在 Header 重写时使用。
- `local_key` 只在本地鉴权时使用。
- 日志、CLI list 输出、错误消息不得展示 secret 明文。
- 详细设计阶段可以定义 secret 掩码策略，例如仅显示长度或固定占位符。

## 11. CLI 行为设计

### 11.1 fas start

流程：

```text
load app config
  → load model_api config
  → validate config
  → build runtime config
  → start server on runtime.app.host:runtime.app.port (默认 127.0.0.1:8787)
```

如果任一启用 profile 配置错误，则拒绝启动。禁用 profile 不阻塞启动。

### 11.2 fas check

流程：

```text
load app config
  → load model_api config
  → validate config
  → print validation result (包含两个默认字段与启用 profile 数)
```

输出应面向人类阅读，指出具体文件、字段路径和错误原因；成功时输出 `Default OpenAI model` 与 `Default Anthropic model` 两行。

### 11.3 fas list

流程：

```text
load config
  → validate enough for listing
  → print profile summary
```

输出字段：

- `model_name`
- `model_id`
- `enabled`
- `type`
- `base_url`
- `default_openai_model` / `default_anthropic_model`（来自 `app.json`）

不得输出真实 `api_key` 或 `local_key` 明文。

## 12. 测试策略

第一版建议以单元测试为主，少量集成测试验证代理链路。

### 12.1 单元测试

- 配置 schema 校验。
- `host` 非空校验（默认 `127.0.0.1`，允许 `localhost` / `0.0.0.0` / 局域网 IP 等任意非空字符串）。
- `local_key`、`default_openai_model` 与 `default_anthropic_model` 必填校验。
- `default_openai_model` / `default_anthropic_model` 分别匹配对应类型启用 profile 的 `model_name` 校验。
- 禁用 profile 不阻塞启动。
- `resolveEndpointFamily` 路径映射（/v1/chat/completions→openai、/v1/messages→anthropic、其他→未支持）。
- profile 匹配逻辑（按 family + model_name）。
- OpenAI 与 Anthropic Header 重写。
- 生成类端点缺省参数补全（`model` 覆盖为 `profile.model_id`）。
- 非生成类端点不改写 body；未支持端点直接 404。
- 日志脱敏函数。

### 12.2 集成测试

- 启动本机服务并请求 mock upstream。
- 验证路径与 query string 原样拼接。
- 验证上游错误状态码与响应体透传。
- 验证流式响应不被完整缓存。
- 验证超时后返回本地错误。

### 12.3 CLI 测试

- `fas check` 对合法配置返回成功。
- `fas check` 对非法配置返回失败。
- `fas list` 输出不包含 secret。
- `fas start` 对非法配置拒绝启动。

## 13. 可扩展性设计

第一版不实现下列能力，但架构上保留扩展点：

- 新接口类型：通过 Header 重写和默认参数策略扩展，而不是改动整个代理流水线。
- 示例配置生成命令：可在 CLI 模块新增 `fas init`，不影响服务模块。
- 自定义配置目录：可在配置加载模块增加路径参数，默认仍使用工程目录 `cfgs`。
- 热重载：未来可在服务层替换 runtime config，但第一版必须启动时读取一次。
- Web UI：未来可复用配置校验模块和 profile 数据结构。
- 桌面托盘：通过独立 `tray` 入口复用核心服务模块，不改变 CLI 与代理流水线。
- token 统计：未来可在代理响应层增加可选统计模块，但第一版不解析模型流内容。

## 14. 关键设计约束

- 不做 OpenAI 与 Claude 协议互转。
- 不通过模型名、端口或 query 参数选择 profile（始终由请求路径的协议家族 + 对应默认字段决定）。
- 不自造 `/v1/models` 模型列表。
- 不自动重试生成类请求。
- 默认只监听 `127.0.0.1`；`host` 可配置为任意非空字符串（`localhost` / `0.0.0.0` / 局域网 IP），但需要用户自行评估安全风险并使用强 `local_key`。
- 不记录 secret 与完整请求/响应正文。
- 不热重载配置（托盘切换默认模型是显式写回 `app.json` 并热更新内存字段，属受控写操作，不属于文件监视式热重载）。
- 不让禁用 profile 参与默认字段匹配。
- 托盘版不得破坏现有 `fas start/check/list` 行为。
- 托盘版不得引入第二套配置查找规则（打包 exe 的 `<exe 同级>/cfgs` 与开发态 `process.cwd()/cfgs` 共用同一份 loader 逻辑，仅通过 `startRuntime` 的 `cfgDir` 参数注入路径）。

## 14.1 下一迭代托盘版约束

- 目标平台：先支持 Windows，架构上保留后续跨平台空间。
- 交付形态：Windows 单文件便携 exe（`electron-builder --win portable`），与 `cfgs/` 目录同级放置即可运行；配套脚本 `scripts/prepare_release_cfgs.mjs` 负责在打包完成后把源码 `cfgs/*.example.json` 拷贝为 `release/cfgs/*.json`（去掉 `.example`），已存在则跳过，且不修改源码工程内的 `cfgs/`。
- 运行模型：托盘入口内置并管理 HTTP 服务，不拉起 `fas start` 子进程。
- 默认体验：静默驻留系统托盘，不显示命令行窗口。
- 调试模式：允许显示调试窗口或控制台信息。
- 菜单范围：右键菜单按 OpenAI / Anthropic 家族分组列出所有 enabled profile，radio 单选切换默认模型（写回 `cfgs/app.json` + 热更新内存 + 系统通知反馈），并保留“退出”项。
- 单实例：重复启动不得创建第二个服务或第二个托盘图标。
- 失败反馈：配置错误、端口占用、服务启动失败通过系统通知反馈；切换默认模型时写文件失败也通过系统通知反馈，不影响进程状态。
- 配置来源：开发态沿用当前 CLI 的配置查找规则（`process.cwd()/cfgs`、`process.cwd()/logs`）；打包 exe 使用 exe 同级目录（`dirname(app.getPath('exe'))/cfgs`、`.../logs`）。

## 15. 待详细设计确认项

以下问题不阻塞当前架构，但需要在后续详细设计和 spec 阶段确认：

1. 目标 Node.js 最低版本。
2. 模块系统选择 ESM 还是 CommonJS。

7. `fas list` 输出格式使用表格、JSON，还是默认表格并提供 JSON 选项。
8. 本地错误 HTTP 状态码与错误码映射。
9. 上游响应 Header 透传的过滤清单。
10. 流式响应在不同 Node.js 版本下的实现细节。
