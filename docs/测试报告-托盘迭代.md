## 5. 托盘冒烟测试

**准备**：临时目录 `%TEMP%\fas-tray-smoke-*\cfgs\` 写入 `app.json`（`port: 18787`）与 `model_api.json`，避免占用生产 8787 端口和已有配置。

### 5.1 托盘启动并拉起 HTTP 服务

```powershell
$env:FAS_TRAY_DEBUG = "1"
electron.exe <项目根> --user-data-dir=<临时目录>
```

观察 stdout：

```
[fas-tray] tray icon: d:\...\assets\icon.ico
{"ts":"...","level":"info","msg":"Server started on http://127.0.0.1:18787"}
{"ts":"...","level":"info","msg":"Enabled profiles: 1"}
{"ts":"...","level":"info","msg":"tray: proxy server started","meta":{"host":"127.0.0.1","port":18787}}
[fas-tray] proxy started on http://127.0.0.1:18787
```

- 图标从 `assets/icon.ico` 加载（回退逻辑保留）。
- 共享 `startRuntime` 调用成功。
- 结果：pass。

### 5.2 HTTP 服务真正生效

```powershell
Invoke-WebRequest http://127.0.0.1:18787/v1/x -Method GET
```

- 返回 HTTP 401（fas 鉴权层拒绝无 local_key 请求）。
- `netstat -ano | Select-String ":18787"` 显示端口 LISTENING。
- 结果：pass。

### 5.3 单实例锁

第二次使用相同 `--user-data-dir` 启动 electron：

- 进程立即退出，退出码 0。
- 端口 18787 依然被首个进程占用，没有产生第二个监听。
- 结果：pass。

### 5.4 启动失败通知路径

故意让 `cfgs/app.json` 内容非法 JSON（BOM 干扰）：

- stdout 输出 `[fas-tray] startRuntime failed: Invalid JSON in file: app.json`。
- 进程保留托盘图标，等待用户手动退出，符合“启动失败不静默失败”的规格。
- 结果：pass（走的是 `notify()` 的错误分支）。

### 5.5 退出释放端口

- 通过 `Stop-Process` 结束进程后，18787 端口不再监听。
- 优雅关闭路径（`RunningRuntime.close()`）在集成测试 §4 用例 1 覆盖：`close()` 后同一端口可被 `net.createServer` 重新监听。
- 结果：pass。
