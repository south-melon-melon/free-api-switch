# Free API Switch 软件详细设计

## 1. 文档定位

本文基于 [需求规格.md](./需求规格.md) 与 [软件架构设计.md](./软件架构设计.md)，对第一版进行软件详细设计，作为后续 spec 与编码实现的直接依据。

本文只覆盖已确认的第一版范围，不引入模型路由、协议互转、热重载、token 统计等非目标能力。

## 2. 技术决策基线

以下决策已确认，详细设计据此展开：

| 项 | 决策 | 说明 |
| --- | --- | --- |
| 运行时 | Node.js LTS 20+ | 稳定支持 ESM、Web Streams、`AbortController`、`node:test` |
| 模块系统 | ESM | `package.json` 设置 `"type": "module"` |
| HTTP 服务 | 原生 `node:http` | 不引入 Web 框架，保持透传语义 |
| 上游客户端 | 原生 `node:http` / `node:https` | 按 `base_url` 协议选择，便于流式透传与 Header 控制 |
| local_key 解析 | 同时支持 `Authorization: Bearer` 与 `x-api-key` | 兼容 OpenAI 与 Anthropic 两类工具默认填法 |
| `fas list` 输出 | 默认表格，支持 `--json` | 表格面向人，JSON 面向脚本 |
| 配置校验库 | `zod` | 统一 schema 校验，服务于 start/check/测试 |
| CLI 解析 | Node.js `node:util` 的 `parseArgs` | 命令简单，不引入额外 CLI 框架 |
| 托盘入口 | Electron 主进程 | 下一迭代使用 Tray、Menu、Notification 和单实例锁实现 Windows 托盘版 |
| 托盘交付 | Windows 单文件便携 exe | `electron-builder --win portable` 产出单个 exe，配合 exe 同级 `cfgs/` 使用 |
| 测试框架 | `node:test` + `node:assert` | 零额外依赖 |

以上决策若与你的偏好不符，可在实现前提出调整。

## 3. 工程与构建

### 3.1 package.json 关键字段

```jsonc
{
  "name": "free-api-switch",
  "version": "0.1.0",
  "type": "module",
  "bin": {
    "fas": "dist/cli/index.js"
  },
  "engines": {
    "node": ">=20"
  },
  "scripts": {
    "build": "tsc -p tsconfig.json",
    "dev": "tsc -w -p tsconfig.json",
    "test": "node --test",
    "check": "tsc --noEmit -p tsconfig.json"
  }
}
```

依赖：

- 运行依赖：`zod`。
- 开发依赖：`typescript`、`@types/node`。
- 下一迭代桌面依赖：`electron` 及 Windows 便携目录打包工具。

托盘版需要新增独立构建脚本，CLI 的 `fas` bin 入口保持不变。实际脚本语义：

```jsonc
{
  "scripts": {
    "build": "tsc -p tsconfig.json",
    "tray:dev": "npm run build && electron .",
    "tray:dist": "npm run build && electron-builder --win portable --config electron-builder.json && node scripts/prepare_release_cfgs.mjs"
  }
}
```

`scripts/prepare_release_cfgs.mjs` 在 `electron-builder` 完成打包后运行，把源码 `cfgs/*.example.json` 拷贝为 `release/cfgs/*.json`（去掉 `.example` 后缀），生成 `release/cfgs/app.json` 与 `release/cfgs/model_api.json`；目标已存在则跳过、不覆盖，也不会修改源码工程内的 `cfgs/`。`.gitignore` 忽略整个 `release/` 目录。

打包产物：`release/Free API Switch <version>.exe` + `release/cfgs/app.json` + `release/cfgs/model_api.json`；exe 首次运行后会在同级自动创建 `logs/`。部署方式：把 exe 与 `cfgs/` 同级放到目标目录（例如 `D:\FAS\`），双击 exe 即可，与 cwd、快捷方式、开机自启无关。

以上脚本名称保持稳定，必须满足：CLI 构建不依赖托盘运行；托盘打包基于已编译的 `dist/tray/main.js`。

## 4. 核心数据结构

### 4.1 types.ts

```ts
export const INTERFACE_TYPES = ['openai_compatible', 'anthropic_compatible'] as const;
export type InterfaceType = (typeof INTERFACE_TYPES)[number];

export const LOG_LEVELS = ['debug', 'info', 'warn', 'error'] as const;
export type LogLevel = (typeof LOG_LEVELS)[number];

/**
 * 抽象推理强度等级到上游真实字段的单一数据源（Single Source of Truth）。
 * 新增/调整等级只需修改本对象，schema、defaults 翻译、UI 下拉框自动跟进。
 */
export const THINKING_LEVEL_MAP = {
  off:    { openai_effort: 'minimal', anthropic_thinking: { type: 'disabled' } },
  low:    { openai_effort: 'low',     anthropic_thinking: { type: 'enabled', budget_tokens: 1024  } },
  medium: { openai_effort: 'medium',  anthropic_thinking: { type: 'enabled', budget_tokens: 4096  } },
  high:   { openai_effort: 'high',    anthropic_thinking: { type: 'enabled', budget_tokens: 12000 } },
  xhigh:  { openai_effort: 'high',    anthropic_thinking: { type: 'enabled', budget_tokens: 24000 } },
} as const;
export type ThinkingLevel = keyof typeof THINKING_LEVEL_MAP;
export const THINKING_LEVELS = Object.keys(THINKING_LEVEL_MAP) as [ThinkingLevel, ...ThinkingLevel[]];

export interface AppConfig {
  host: string;
  port: number;
  log_level: LogLevel;
  request_timeout_ms: number;
  local_key: string;
  default_openai_model: string;
  default_anthropic_model: string;
}

export interface ProfileConfig {
  model_name: string;   // 本地标识：用于与 app.default_openai_model / default_anthropic_model 匹配
  model_id: string;     // 真正下发到上游 API 的模型名（写回 body.model）
  enabled: boolean;
  type: InterfaceType;
  base_url: string;
  api_key: string;
  max_output_tokens?: number;
  max_context_tokens?: number;
  /** 抽象推理强度等级，请求缺 thinking_level 时按 profile.type 翻译并注入 */
  thinking_level?: ThinkingLevel;
  defaults: Record<string, unknown>;
}

export type EndpointFamily = 'openai' | 'anthropic';

export interface RuntimeConfig {
  app: AppConfig;
  profiles: ProfileConfig[];
  /** 启用 profile 按 model_name 建立的索引，用于按端点家族路由查找 */
  enabledProfilesByModelName: Map<string, ProfileConfig>;
}
```

说明：

- `AppConfig.host` 为任意非空字符串（默认 `127.0.0.1`；支持 `localhost` / `0.0.0.0` / 局域网 IP）。
- `AppConfig` 新增 `local_key`、`default_openai_model`、`default_anthropic_model` 三个字段。`local_key` 为全局统一的本地鉴权令牌；`default_openai_model` 决定 `/v1/chat/completions` 走哪个 profile；`default_anthropic_model` 决定 `/v1/messages` 走哪个 profile。
- `ProfileConfig` 的 `model_name` 是本地标识（用于匹配两个默认字段），`model_id` 是真正下发到上游 API 的模型名。请求体中的 `model` 字段将始终被覆盖为 `model_id`。
- `max_output_tokens` 和 `max_context_tokens` 为可选字段。设置后，请求缺失对应字段时优先使用此值，其次使用 `defaults` 中的对应值。
- `thinking_level` 为可选抽象等级（`off / low / medium / high / xhigh`），代理转发时根据 `profile.type` 通过 `THINKING_LEVEL_MAP` 翻译为上游真实字段（详见 §7.2）。
- `defaults` 用 `Record<string, unknown>`，第一版不对内部字段做类型约束，只做"缺失即补全"。
- `enabledProfilesByModelName` 在校验通过后一次性构建，运行期只读；`resolveProfile(runtime, localKey, family)` 按 family 对应默认字段读取此索引。

## 5. 配置模块详细设计

### 5.1 schema.ts

```ts
import { z } from 'zod';
import { INTERFACE_TYPES, LOG_LEVELS, THINKING_LEVELS } from '../core/types.js';

export const appSchema = z.object({
  host: z.string().min(1),
  port: z.number().int().min(1).max(65535),
  log_level: z.enum(LOG_LEVELS),
  request_timeout_ms: z.number().int().positive(),
  local_key: z.string().min(1),
  default_openai_model: z.string().min(1),
  default_anthropic_model: z.string().min(1),
}).strict();

export const profileSchema = z.object({
  model_name: z.string().min(1),
  model_id: z.string().min(1),
  enabled: z.boolean().default(true),
  type: z.enum(INTERFACE_TYPES),
  base_url: z.string().url(),
  api_key: z.string().min(1),
  max_output_tokens: z.number().int().positive().optional(),
  max_context_tokens: z.number().int().positive().optional(),
  thinking_level: z.enum(THINKING_LEVELS).optional(),
  defaults: z.record(z.unknown()).default({}),
}).strict();

export const modelApiSchema = z.object({
  profiles: z.array(profileSchema),
}).strict();
```

schema 层负责结构与基础类型校验；跨记录的唯一性、`base_url` 语义以及两个默认字段与 profile `model_name` + `type` 的匹配关系交由业务级校验，以便产出更精确的错误定位。`host` 只做非空校验，不再限制字面量，允许 `127.0.0.1` / `localhost` / `0.0.0.0` / 局域网 IP 等值。

### 5.2 loadConfig.ts

```ts
export interface LoadedRawConfig {
  appRaw: unknown;
  modelApiRaw: unknown;
}

export function loadRawConfig(cfgDir: string): LoadedRawConfig;
```

行为：

- 依次读取 `cfgDir/app.json`、`cfgDir/model_api.json`。
- 文件不存在 → 抛 `LocalError(CONFIG_FILE_NOT_FOUND)`，消息包含缺失文件路径。
- `JSON.parse` 失败 → 抛 `LocalError(CONFIG_JSON_INVALID)`，消息包含出错文件名，不含文件全文。
- 不做 schema 校验，只负责 IO 与 JSON 解析。

### 5.3 validateConfig.ts

```ts
export function validateConfig(raw: LoadedRawConfig): RuntimeConfig;
```

流程：

1. `appSchema.safeParse(raw.appRaw)`，失败 → `CONFIG_VALIDATION_FAILED`，附带 zod issue 路径。
2. `modelApiSchema.safeParse(raw.modelApiRaw)`，失败同上。
3. 业务级校验（全量视角，见下）。
4. 按 `model_name` 为启用 profile 构建 `enabledProfilesByModelName` 索引，同时校验 `default_openai_model` / `default_anthropic_model` 命中的 profile 类型与家族一致。
5. 返回 `RuntimeConfig`。

业务级校验规则（严格校验，任一失败即拒绝）：

| 规则 | 范围 | 错误定位 |
| --- | --- | --- |
| `local_key` 非空 | app | 缺失时拒绝 |
| `default_openai_model` 与 `default_anthropic_model` 均非空 | app | 缺失时拒绝 |
| `model_name` 唯一 | 全部 profile | 重复的 model_name |
| 存在 `model_name` 等于 `default_openai_model` 且 `type=openai_compatible` 的启用 profile | 全部 profile | 无匹配 / 类型不符时拒绝 |
| 存在 `model_name` 等于 `default_anthropic_model` 且 `type=anthropic_compatible` 的启用 profile | 全部 profile | 无匹配 / 类型不符时拒绝 |
| `base_url` 不以 `/v1` 结尾、不含 `/v1` 段 | 仅 enabled profile | 违规 profile model_name |
| `type` 合法 | 仅 enabled profile | 已由 schema 保证，双重兜底 |
| `api_key` 与 `model_id` 非空 | 仅 enabled profile | 已由 schema 保证 |

`enabled: false` 的 profile：不参与默认字段匹配，也不阻塞服务启动。

`base_url` 校验判定：解析 URL 后，若 `pathname` 包含以 `/v1` 开头的段（如 `/v1`、`/v1/`），判为违规。需求明确 `base_url` 不含 `/v1`。

## 6. Auth 与 Profile 解析模块详细设计

### 6.1 extractLocalKey

```ts
export function extractLocalKey(headers: import('node:http').IncomingHttpHeaders): string | undefined;
```

解析优先级：

1. `authorization` 头存在且形如 `Bearer <token>` → 取 `<token>`。
2. 否则 `x-api-key` 头存在 → 取其值。
3. 都没有 → 返回 `undefined`。

说明：`Bearer` 前缀不区分大小写；去除首尾空白；空字符串按未提供处理。

### 6.2 validateLocalKey

```ts
export function validateLocalKey(
  runtime: RuntimeConfig,
  localKey: string | undefined,
): void;
```

行为：

- `localKey` 为空 → 抛 `LocalError(LOCAL_KEY_MISSING)`。
- `localKey` 不等于 `runtime.app.local_key` → 抛 `LocalError(INVALID_LOCAL_KEY)`。
- 匹配成功 → 返回 void。

### 6.3 resolveEndpointFamily 与 resolveProfile

```ts
export type EndpointFamily = 'openai' | 'anthropic';

export function resolveEndpointFamily(pathname: string): EndpointFamily | undefined;

export function resolveProfile(
  runtime: RuntimeConfig,
  localKey: string,
  family: EndpointFamily,
): ProfileConfig;
```

`resolveEndpointFamily` 行为：

- `/v1/chat/completions` → `'openai'`。
- `/v1/messages` → `'anthropic'`。
- 其他路径 → `undefined`（调用侧据此返回 HTTP 404 `UNSUPPORTED_ENDPOINT`，不落到上游）。

`resolveProfile` 行为：

- 根据 `family` 选择默认字段：`openai` → `runtime.app.default_openai_model`；`anthropic` → `runtime.app.default_anthropic_model`。
- 通过默认字段值在 `runtime.enabledProfilesByModelName` 中查找命中的启用 profile。
- 命中 → 返回 profile。
- 未命中 → 抛 `LocalError(PROFILE_NOT_FOUND)`。

说明：此函数在鉴权通过、端点家族识别后调用。`localKey` 参数保留供未来按调用者做审计或多租户扩展；第一版仅用于日志/元数据。由于配置校验阶段已确保两个默认字段能匹配到对应类型的启用 profile，运行期正常情况不会触发 `PROFILE_NOT_FOUND`，该错误码主要用于防御性编程。

## 7. 参数补全模块详细设计

### 7.1 端点判定

```ts
const GENERATION_ENDPOINTS = ['/v1/chat/completions', '/v1/messages'] as const;

export function isGenerationEndpoint(pathname: string): boolean;
```

判定基于请求路径的 `pathname`（忽略 query），精确匹配上述路径。等价地，`resolveEndpointFamily(pathname) !== undefined` 即为生成类端点；其他生成类端点直接返回 HTTP 404 `UNSUPPORTED_ENDPOINT`。

### 7.2 applyDefaults

```ts
export function applyDefaults(
  bodyText: string,
  profile: ProfileConfig,
): { bodyText: string; appliedDefaults: boolean };
```

行为：

1. 若请求不是生成类端点 → 调用方不会调用本函数，body 原样透传。
2. 生成类端点：
   - `JSON.parse(bodyText)`，失败 → 抛 `LocalError(REQUEST_BODY_INVALID_JSON)`。
   - 解析结果非对象（数组/标量）→ 同样抛 `REQUEST_BODY_INVALID_JSON`。
   - `model`：始终覆盖为 `profile.model_id`（真正下发到上游 API 的模型名）。
   - `max_tokens`：若缺失 → 优先用 `profile.max_output_tokens`，其次用 `profile.defaults.max_tokens`；已有值不覆盖。
   - `max_context_tokens`：若缺失 → 优先用 `profile.max_context_tokens`，其次用 `profile.defaults.max_context_tokens`；已有值不覆盖。
   - **抽象推理强度 `thinking_level`**（本项目自定义字段，不透传上游）：
     - 若 body 已带上游真实字段（`openai_compatible`：`reasoning_effort` 或 `reasoning` 对象；`anthropic_compatible`：`thinking` 对象）→ 完全透传，不做任何注入。
     - 否则按优先级取抽象等级：`body.thinking_level` > `profile.thinking_level`；取到的等级通过 `THINKING_LEVEL_MAP` 查表：
       - `openai_compatible` → 写入 `body.reasoning_effort = mapping.openai_effort`。
       - `anthropic_compatible` → 写入 `body.thinking = { ...mapping.anthropic_thinking }`（`off` 时值为 `{ type: 'disabled' }`）。
     - 无论是否注入，最终都从 body 中 `delete thinking_level`，保证上游收不到未知字段。
     - `body.thinking_level` 只接受 `THINKING_LEVELS` 中的合法值，非法值忽略并回退到 profile 等级。
   - 遍历 `profile.defaults`，仅补 body 中缺失的顶层字段（`!(key in body)`），跳过已由上述专用字段处理的 key（`max_tokens`、`max_context_tokens`、`thinking_level`）。
   - `appliedDefaults` 记录是否发生任一补全，供日志使用。
   - 重新 `JSON.stringify`，返回新 body。

补全只作用于顶层字段；不做 OpenAI/Claude 字段互转；不递归。

### 7.3 空 body 处理

生成类端点若请求 body 为空字符串：视为缺失合法 JSON，按 `REQUEST_BODY_INVALID_JSON` 处理。此点在第 12 节确认项列出，待你确认是否改为"空 body 直接透传"。

## 8. 代理与转发模块详细设计

### 8.1 buildTargetUrl

```ts
export function buildTargetUrl(baseUrl: string, reqUrl: string): URL;
```

行为：

- `reqUrl` 为 Node 原生 `req.url`，形如 `/v1/chat/completions?foo=bar`。
- 将 `reqUrl`（含 path 与 query）原样拼到 `baseUrl` 之后。
- 规则：`new URL(baseUrl)` 的 `pathname` 末尾去重斜杠后 + `reqUrl`。
- 保留 query string 原样。

示例：`base_url=https://api.deepseek.com` + `/v1/chat/completions` → `https://api.deepseek.com/v1/chat/completions`。

### 8.2 buildUpstreamHeaders

```ts
export function buildUpstreamHeaders(
  incoming: import('node:http').IncomingHttpHeaders,
  profile: ProfileConfig,
): Record<string, string>;
```

行为：

1. 复制 incoming headers 作为基础。
2. 删除本地鉴权字段：`authorization`、`x-api-key`。
3. 删除 hop-by-hop 及连接类 header：`connection`、`keep-alive`、`proxy-authorization`、`proxy-authenticate`、`te`、`trailer`、`transfer-encoding`、`upgrade`。
4. 删除 `host`（由上游客户端按 target host 重新设置）。
5. 删除 `content-length`（body 可能因补全变化，交由客户端按实际 body 重算）。
6. 按 profile 类型注入上游鉴权：
   - `openai_compatible` → `Authorization: Bearer <api_key>`。
   - `anthropic_compatible` → `x-api-key: <api_key>`。
7. 其余业务 header（如 `content-type`、`accept`、`anthropic-version` 等）保留透传。

注：`anthropic_version` 属于请求体 `defaults` 中的字段（见需求示例），非 header，二者不冲突。

### 8.3 proxyToUpstream

```ts
export interface ProxyInput {
  target: URL;
  method: string;
  headers: Record<string, string>;
  bodyText: string | null;     // 生成类端点为改写后 body；其余端点为 null（直接管道透传原始流）
  rawReq: import('node:http').IncomingMessage;
  res: import('node:http').ServerResponse;
  timeoutMs: number;
  logger: Logger;
}

export function proxyToUpstream(input: ProxyInput): Promise<void>;
```

行为：

1. 按 `target.protocol` 选择 `node:http` 或 `node:https` 的 `request`。
2. 设置 `AbortController`，`timeoutMs` 到时 abort，抛/响应 `UPSTREAM_TIMEOUT`。
3. body 处理：
   - `bodyText !== null`（生成类端点已读取并可能改写）：设置 `content-length` 为 UTF-8 字节长度，写入 body。
   - `bodyText === null`（非生成类端点）：将 `rawReq` 以流方式 `pipe` 到上游请求，不缓存完整 body。
4. 上游响应：
   - 写回上游 `statusCode` 与经过滤的响应 header（删除 hop-by-hop 类）。
   - 将上游响应流 `pipe` 到 `res`，实现流式透传，不缓存完整响应。
5. 连接错误（ECONNREFUSED、DNS 失败等）→ `UPSTREAM_CONNECTION_FAILED`。
6. 超时 → `UPSTREAM_TIMEOUT`。
7. 上游返回的 HTTP 错误（4xx/5xx）不视为本地错误，状态码与响应体原样透传。

响应 header 过滤：删除 `connection`、`keep-alive`、`transfer-encoding`、`proxy-authenticate`、`trailer`、`upgrade`；其余原样透传（含 `content-type`、`content-length`、SSE 相关）。

## 9. HTTP 服务模块详细设计

### 9.1 startServer

```ts
export function startServer(runtime: RuntimeConfig, logger: Logger): Promise<import('node:http').Server>;
```

- 使用 `http.createServer(handler)`。
- `server.listen(runtime.app.port, runtime.app.host)`，绑定地址来自配置（默认 `127.0.0.1`，允许任意非空字符串，如 `localhost` / `0.0.0.0` / 局域网 IP）；配置侧只做非空校验，不再强制字面量为 `127.0.0.1`。
- 监听 `error` 事件：端口占用（EADDRINUSE）等启动错误向 CLI 抛出并使进程以非零码退出。
- 成功监听后打印启动信息（host、port、启用 profile 数量、`default_openai_model` 与 `default_anthropic_model`），不含 secret。
- 下一迭代需要返回可关闭的 server handle，供 CLI 信号处理和托盘“退出”菜单调用。

下一迭代新增共享启动函数：

```ts
export interface StartRuntimeOptions {
  /** cfgs 目录路径，缺省使用 process.cwd()/cfgs */
  cfgDir?: string;
  /** logs 目录路径，缺省使用 process.cwd()/logs */
  logDir?: string;
}

export interface RunningRuntime {
  runtime: RuntimeConfig;
  logger: Logger;
  /** cfgs 目录的绝对路径；托盘切换默认模型时定位 app.json */
  cfgDir: string;
  logDir: string;
  server: import('node:http').Server;
  /** 优雅关闭 HTTP 服务 */
  close(): Promise<void>;
}

export function startRuntime(options?: StartRuntimeOptions): Promise<RunningRuntime>;
```

流程：

```text
loadRawConfig(cfgDir) → validateConfig → createLogger(app.log_level, logDir)
→ startServer(runtime, logger, logDir) → RunningRuntime { runtime, logger, cfgDir, logDir, server, close }
```

`runStart` 和托盘入口都调用 `startRuntime`。CLI 使用默认路径（`process.cwd()/cfgs`、`process.cwd()/logs`）；托盘入口在 `app.isPackaged === true` 分支下显式传入 `cfgDir = <dirname(app.getPath('exe'))>/cfgs` 与 `logDir = <dirname(app.getPath('exe'))>/logs`，开发态（`npm run tray:dev`）保持缺省。CLI 负责把错误打印到控制台；托盘入口负责把错误显示为系统通知。托盘还依赖 `RunningRuntime.cfgDir` 定位 `cfgs/app.json` 做默认模型写回。

### 9.2 handleRequest 流水线

```ts
export function createRequestHandler(runtime: RuntimeConfig, logger: Logger): http.RequestListener;
```

单请求处理顺序：

```text
start = now()
try:
  localKey    = extractLocalKey(req.headers)
  validateLocalKey(runtime, localKey)
  family      = resolveEndpointFamily(new URL(req.url, base).pathname)
  if family is undefined:
      writeLocalError(res, new LocalError('UNSUPPORTED_ENDPOINT', ...))   # HTTP 404，不落上游
      return
  profile     = resolveProfile(runtime, localKey, family)
  target      = buildTargetUrl(profile.base_url, req.url)
  isGen       = isGenerationEndpoint(target.pathname)                     # 与 family 等价，双重兜底
  appliedDefaults = false
  if isGen:
      bodyText = await readBody(req)                # 完整读取
      ({ bodyText, appliedDefaults } = applyDefaults(bodyText, profile))
  else:
      bodyText = null                               # 直接流式透传（未支持路径已在上方拦截）
  headers     = buildUpstreamHeaders(req.headers, profile)
  await proxyToUpstream({ target, method: req.method, headers, bodyText, rawReq: req, res, timeoutMs, logger })
catch LocalError as e:
  writeLocalError(res, e)
catch unknown as e:
  writeLocalError(res, new LocalError('INTERNAL_ERROR', ...))
finally:
  logger.info('access', { profile_model_name, model_id, type, family, path, base_url, status, duration_ms, applied_defaults })
  if isGen and (requestMessages or responseMessage):
      writeMessageLog(logDir, requestMessages, responseMessage)   # 独立文件：logs/messages/<ts>.json
```

`readBody`：将请求流聚合为字符串，仅用于生成类端点。上限为 10MB（10 * 1024 * 1024 字节，按累计字节数判定），超限抛 `LocalError(PAYLOAD_TOO_LARGE)` → HTTP 413（已确认）。

## 10. 日志与脱敏详细设计

### 10.1 logger.ts（运行日志：控制台 + 文件）

```ts
export interface Logger {
  debug(msg: string, meta?: Record<string, unknown>): void;
  info(msg: string, meta?: Record<string, unknown>): void;
  warn(msg: string, meta?: Record<string, unknown>): void;
  error(msg: string, meta?: Record<string, unknown>): void;
}

export function createLogger(level: LogLevel, logDir?: string): Logger;
```

- 级别按 `debug < info < warn < error` 过滤。
- 输出到 stdout/stderr（error 走 stderr）。
- 同时写入 `logDir` 下的 `fas-YYYY-MM-DD.log`，按天轮转，JSON-line 格式（每行一个 JSON 对象）。
- 日志目录不存在时自动创建。
- `meta` 在输出前统一经过 `redactMeta`。

### 10.2 消息日志（每次调用独立文件）

行为：

- `requestHandler` 在生成类端点结束后，将本次调用的完整请求 messages 与助手响应消息写入独立文件。
- 路径：`logs/messages/<ISO 时间戳>.json`。文件名中冒号和点会替换为 `-` 以兼容文件系统。
- **请求侧**：`request_messages` 是本次实际发送给上游 API 的完整 `messages` 数组，包含 system prompt、上层软件插入的提示词、历史对话与本次用户输入。
- **响应侧**：`response_message` 是本次上游返回的助手消息（单条）。
- 文件内容示例：

```json
{
  "ts": "2026-07-04T00:52:04.266Z",
  "request_messages": [
    { "role": "system", "content": "……" },
    { "role": "user", "content": "……" },
    { "role": "assistant", "content": "……" },
    { "role": "user", "content": "……本次用户输入……" }
  ],
  "response_message": {
    "role": "assistant",
    "content": "……本次助手回复……",
    "reasoning_content": "……思考过程（若有）……",
    "finish_reason": "stop"
  }
}
```

- `request_messages` 由 `extractRequestMessages` 解析请求 body 得到（`body.messages` 数组原样保留）。
- `response_message` 由 `extractAssistantMessage` 解析响应 body 得到：
  - 非流式 JSON：OpenAI 兼容取 `choices[0].message`；Anthropic 兼容组装为 `{ role, content }`。
  - SSE 流式响应：`aggregateSseResponse` 按 `profile.type` 聚合 delta 为单个 message（OpenAI 累积 `delta.content` / `reasoning_content` / `tool_calls`；Anthropic 累积 `content_block_start` + `content_block_delta` 中的 `text_delta` / `input_json_delta` / `thinking_delta`，并合并 `message_delta.stop_reason`）。
- 消息日志写入失败不阻塞主流程。

### 10.3 访问日志字段（allowed in fas-YYYY-MM-DD.log）

```ts
interface AccessLog {
  ts: string;              // ISO 时间
  profile_model_name: string;  // profile.model_name
  model_id: string;            // profile.model_id
  type: InterfaceType;
  family: EndpointFamily;      // openai / anthropic
  path: string;            // 仅 pathname，剔除 query（已确认）
  base_url: string;
  status: number;
  duration_ms: number;
  applied_defaults: boolean;
}
```

### 10.4 redact.ts

- 提供 `redactHeaders`：将 `authorization`、`x-api-key`、`cookie` 值替换为 `***`。
- 提供 `redactMeta`：对运行日志对象中的敏感字段做脱敏处理。
- `requestHandler` 内的 `extractRequestMessages` 从请求 body 中提取 messages 数组。
- `requestHandler` 内的 `extractResponseContent` 从响应 body 中提取结构化内容（OpenAI 兼容 `choices[]`、Anthropic 兼容 `content[]`）。
- 禁止记录：真实 `api_key`、`local_key` 明文、鉴权头明文、Cookie 明文、未经脱敏的完整请求/响应正文。
- 消息日志中的 `request_messages` / `response_content` 直接由 `requestHandler` 落入 `logs/messages/`，不进入运行日志。

## 11. 错误处理详细设计

### 11.1 LocalError 与错误码

```ts
export type ErrorCode =
  | 'CONFIG_FILE_NOT_FOUND'
  | 'CONFIG_JSON_INVALID'
  | 'CONFIG_VALIDATION_FAILED'
  | 'LOCAL_KEY_MISSING'
  | 'INVALID_LOCAL_KEY'
  | 'UNSUPPORTED_ENDPOINT'
  | 'PROFILE_NOT_FOUND'
  | 'PROFILE_DISABLED'
  | 'REQUEST_BODY_INVALID_JSON'
  | 'PAYLOAD_TOO_LARGE'
  | 'UPSTREAM_CONNECTION_FAILED'
  | 'UPSTREAM_TIMEOUT'
  | 'INTERNAL_ERROR';

export class LocalError extends Error {
  constructor(public code: ErrorCode, message: string) { super(message); }
}
```

### 11.2 HTTP 状态码映射（本地错误）

| 错误码 | HTTP 状态 | 触发场景 |
| --- | --- | --- |
| `LOCAL_KEY_MISSING` | 401 | 请求未携带可解析的 local_key |
| `INVALID_LOCAL_KEY` | 401 | local_key 与 `app.local_key` 不匹配 |
| `UNSUPPORTED_ENDPOINT` | 404 | 请求路径未匹配任何生成类端点家族，不落上游 |
| `PROFILE_NOT_FOUND` | 401 | 家族对应默认字段未匹配到启用 profile（防御性） |
| `PROFILE_DISABLED` | 401 | 保留（第一版运行期不单独触发） |
| `REQUEST_BODY_INVALID_JSON` | 400 | 生成类端点 body 非合法 JSON 对象（含空 body） |
| `PAYLOAD_TOO_LARGE` | 413 | 生成类端点请求 body 超过 10MB |
| `UPSTREAM_CONNECTION_FAILED` | 502 | 无法连接上游 |
| `UPSTREAM_TIMEOUT` | 504 | 上游响应超时 |
| `INTERNAL_ERROR` | 500 | 未预期异常 |

配置类错误（`CONFIG_*`）只在 CLI 启动/检查阶段出现，不作为 HTTP 响应，而是打印到终端并以非零退出码结束。

### 11.3 本地错误响应体

```json
{
  "error": {
    "code": "INVALID_LOCAL_KEY",
    "message": "Invalid local API key"
  }
}
```

`message` 为固定英文短语或安全描述，不回显 secret、请求正文或上游响应正文。

### 11.4 上游错误

上游返回 HTTP 响应即视为"成功转发"，状态码与响应体原样透传，不改写、不包装。

## 12. 已确认的边界决策

以下五项曾为待确认项，现已确认：

1. 禁用 profile 的运行期表现：默认字段匹配时跳过禁用 profile，若无其他匹配则返回 `PROFILE_NOT_FOUND`。
2. 生成类端点空 body：按 `REQUEST_BODY_INVALID_JSON`（HTTP 400）处理。
3. 访问日志 `path` 字段：仅记录 pathname，剔除 query。
4. 生成类端点请求 body 上限：10MB，超限返回 `PAYLOAD_TOO_LARGE`（HTTP 413）。
5. `fas list --json` 输出结构：`{ "app": { "local_key": "***", "default_openai_model": "deepseek-v4-flash", "default_anthropic_model": "claude-sonnet-4-6" }, "profiles": [ { model_name, model_id, enabled, type, base_url } ] }`；表格模式按固定列顺序左对齐输出。

## 15. Windows 托盘版详细设计

### 15.1 tray/main.ts

职责：

- Electron `app` ready 后创建托盘图标。
- 调用 `app.requestSingleInstanceLock()` 获取单实例锁。
- 单实例锁获取失败时直接退出，不启动第二个 HTTP 服务。
- 判断 `app.isPackaged`：为 `true`（打包 exe）时向 `startRuntime` 传入 `cfgDir = dirname(app.getPath('exe'))/cfgs`、`logDir = dirname(app.getPath('exe'))/logs`；为 `false`（`npm run tray:dev`）时使用缺省，与 CLI 保持一致。
- 调用 `startRuntime({ cfgDir?, logDir? })` 启动 fas 本机代理服务。
- 创建托盘右键菜单：按 OpenAI / Anthropic 家族分组列出所有 enabled profile（radio 单选，当前默认模型打勾）；分组下若无匹配 profile，显示灰色 `(无 enabled openai profile)` 或 `(无 enabled anthropic profile)`；菜单末尾始终有“退出”项。启动后 runtime 就绪时重建一次菜单以填充 profile 列表。
- 用户点击某个模型项时，执行 15.5 描述的切换默认模型交互。
- 启动失败时通过 `Notification` 发送系统通知，并保留托盘图标。
- 用户点击“退出”时关闭 HTTP server，再退出 Electron 应用。

### 15.2 托盘启动流程

```text
Electron app ready
  → requestSingleInstanceLock
  → create Tray
  → build initial context menu（此时 running 尚未就绪，两个家族均显示灰色占位）
  → 判断 app.isPackaged：
      true  → startRuntime({ cfgDir: dirname(exe)/cfgs, logDir: dirname(exe)/logs })
      false → startRuntime()
  → success: 写启动日志 + rebuild 菜单填充 profile 列表 + 更新 tray tooltip
  → failure: 系统通知 + 写错误日志（保留托盘图标）
```

默认不创建 BrowserWindow，不显示命令行窗口。

### 15.3 调试模式

托盘版需要保留调试模式入口。实现方式可使用命令行参数或环境变量，例如：

```text
fas-tray.exe --debug
FAS_TRAY_DEBUG=1
```

调试模式行为：

- 可以打开调试窗口或显示控制台信息。
- 不改变配置查找规则。
- 不改变代理服务端口和请求处理行为。
- 错误仍写入运行日志。

### 15.5 切换默认模型交互

托盘右键菜单点击某个 profile radio 项时执行：

```text
switchDefaultModel(family, modelName)
  → field = family === 'openai' ? 'default_openai_model' : 'default_anthropic_model'
  → appJsonPath = resolve(running.cfgDir, 'app.json')
  → 读取 app.json（readFileSync utf-8）
  → JSON.parse → 设置 parsed[field] = modelName
  → writeFileSync(appJsonPath, JSON.stringify(parsed, null, 2) + '\n', 'utf-8')
  → 更新内存：running.runtime.app[field] = modelName（下一次请求即生效）
  → Notification: "已切换 {family} 默认模型：{modelName}"
  → rebuildTrayMenu()  # 刷新 radio 勾选
```

失败处理：

- 若读文件、`JSON.parse` 或写回失败：写运行日志（error 级）+ `Notification("切换失败", "写入 app.json 失败：{msg}")`，**不修改** `running.runtime.app`，也不重建菜单。
- 切换过程不重启 HTTP server，不重新加载 `model_api.json`。

### 15.6 退出流程

```text
用户右键托盘图标 → 退出
  → 禁止新的退出重复触发
  → 如果 HTTP 服务已启动，调用 running.close() 释放监听端口
  → 写入退出日志
  → app.quit()
```

退出流程必须释放监听端口，避免下一次启动出现端口占用。

### 15.7 失败反馈

需要通过系统通知反馈的场景：

- 配置文件不存在。
- 配置 JSON 格式错误。
- 配置校验失败。
- 端口已被占用。
- HTTP 服务启动失败。
- 切换默认模型时写回 `cfgs/app.json` 失败。

通知内容不得包含 `api_key`、`local_key`、Authorization、Cookie 或完整请求/响应正文。

### 15.8 打包输出

Windows 托盘版输出为单文件便携 exe：

```text
release/
├─ Free API Switch <version>.exe   # electron-builder --win portable 产物
└─ cfgs/                            # scripts/prepare_release_cfgs.mjs 从 cfgs/*.example.json 拷贝
   ├─ app.json
   └─ model_api.json
# exe 首次运行后会在同级自动生成 logs/
```

`scripts/prepare_release_cfgs.mjs` 在 `electron-builder` 之后运行，将源码 `cfgs/*.example.json` 复制为 `release/cfgs/*.json`（去掉 `.example`），目标已存在则跳过、不覆盖，也不会修改源码工程内的 `cfgs/`。`package.json` 的 `tray:dist` 脚本为：`npm run build && electron-builder --win portable --config electron-builder.json && node scripts/prepare_release_cfgs.mjs`。`.gitignore` 忽略整个 `release/` 目录。

部署方式：把 exe 与 `cfgs/` 同级放到任意目录（例如 `D:\FAS\`），双击即可运行，与工作目录、快捷方式、开机自启无关；打包 exe 通过 `app.isPackaged === true` 分支从 `dirname(app.getPath('exe'))/cfgs` 读取配置、向 `dirname(app.getPath('exe'))/logs` 写日志。

第一版托盘交付不要求安装器、不要求开机自启动、不要求开始菜单快捷方式。

### 15.9 配置管理窗口

托盘版提供一个内置的配置管理窗口，用于在不手工编辑 JSON 的前提下完整可视化编辑 `cfgs/app.json` 与 `cfgs/model_api.json`。

入口：

- 托盘右键菜单新增"配置管理..."项，位于家族分组与"退出"之间。
- 点击后创建一个 `BrowserWindow`（若已存在则聚焦），窗口关闭时销毁，不影响 HTTP 服务与托盘图标。默认 `window-all-closed` 事件被覆盖，关闭配置窗口不会退出应用。

技术选型：

- 不引入 React/Vue 等前端框架，使用内联 HTML/CSS/TypeScript（`src/tray/settingsAssets.ts`）+ Electron preload 脚本（IPC bridge）；打包时将 HTML 与 preload 以字符串常量形式内联到 `dist/tray/settingsAssets.js`，运行时通过 `materializeSettingsAssets()` 释放到 `userData/settings-ui/`。写盘前将 `__THINKING_LEVELS__` 占位符替换为主进程侧的 `JSON.stringify(THINKING_LEVELS)`，保证 UI 下拉框与 `core/types.ts` 单一数据源一致。

窗口职责：

- 左侧两个 tab：`应用配置`（app.json）与`模型 API`（model_api.json）。
- app 表单字段：`host` / `port` / `log_level` / `request_timeout_ms` / `local_key` / `default_openai_model` / `default_anthropic_model`；默认模型两个下拉框按 `type` + `enabled` 过滤备选项。
- profiles 表单：左侧列表支持新增 / 复制 / 删除（删除需二次确认）；右侧详情表单编辑 `model_name / model_id / enabled / type / base_url / api_key（明文）/ max_output_tokens / max_context_tokens / thinking_level（下拉：off/low/medium/high/xhigh）/ defaults（JSON 文本框）`。

IPC 交互：

- `fas:settings:load` → 主进程读取 cfgDir 下两份 JSON 返回给渲染层。
- `fas:settings:save` → 主进程执行 §15.10 保存流程；渲染层根据返回结果显示成功或错误信息。

保存流程（IPC 主进程侧）：

1. 使用当前 `runtime.app` 的 schema 复用 `validateConfig` 对客户端传入的两份 JSON 做 schema + 业务级校验（例如 `default_openai_model` 必须命中启用的 openai_compatible profile）。
2. 校验失败 → 返回错误 payload，不写盘、不重启服务，UI 显示错误行。
3. 校验通过 → 以 `.tmp -> rename` 原子替换 `cfgs/app.json` 与 `cfgs/model_api.json`（缩进 2、UTF-8）。
4. 调用 `restartRuntime()`：关闭旧 server → 使用同一 `cfgDir` / `logDir` 重新 `startRuntime` → 刷新托盘 tooltip 与右键菜单。
5. 通过 IPC 返回成功。

约束：

- 保存动作是唯一改写配置文件的路径（切换默认模型走 15.5 的独立写回路径）。
- `local_key` / `api_key` 直接明文展示与编辑，第一版不做掩码；日志脱敏规则不变，敏感字段不进入运行日志。
- 保存后重启是"关闭并重新监听"，`host` / `port` 变更会立即生效；若新端口被占用则重启失败，通过 IPC 返回错误。

## 16. 测试用例映射

在架构文档测试策略基础上，明确关键断言：

- 配置：`host` 空字符串被拒（非空即通过；不再强制 `127.0.0.1`）；`local_key` 缺失被拒；`default_openai_model` / `default_anthropic_model` 任一缺失被拒；两个默认字段必须分别匹配对应类型（openai_compatible / anthropic_compatible）的启用 profile 的 `model_name`，否则被拒；禁用 profile 不阻塞启动；`base_url` 含 `/v1` 被拒。
- 鉴权：`Authorization: Bearer` 与 `x-api-key` 均能取到 token；token 与 `app.local_key` 不匹配 → `INVALID_LOCAL_KEY`；缺失 → `LOCAL_KEY_MISSING`。
- 路由：`resolveEndpointFamily('/v1/chat/completions')` → `openai`；`resolveEndpointFamily('/v1/messages')` → `anthropic`；其他路径 → `undefined`，走 HTTP 404 `UNSUPPORTED_ENDPOINT` 且不落上游。
- 解析：`app.default_openai_model` 与 `app.default_anthropic_model` 分别通过 `enabledProfilesByModelName` 命中对应 profile；禁用 profile 被跳过。
- 补全：model 始终覆盖为 `profile.model_id`；max_tokens 缺失时优先用 max_output_tokens；max_context_tokens 缺失时优先用 max_context_tokens；thinking_level 缺失时优先用 profile.thinking_level；已有字段不覆盖（model 除外）；非生成端点 body 不被解析；非法 JSON → 400。
- Header：OpenAI 生成 `Authorization: Bearer`；Anthropic 生成 `x-api-key`；local 鉴权与 hop-by-hop 被删除。
- 代理：path+query 原样拼接；上游 4xx/5xx 状态码与 body 透传；超时 → 504；连接失败 → 502。
- 脱敏：redactHeaders 屏蔽鉴权/Cookie；运行日志不含 body 内容。
- 文件日志：验证 `logs/fas-YYYY-MM-DD.log` 文件创建，内容为 JSON-line 格式。
- 消息日志：生成类端点调用后，验证 `logs/messages/<ts>.json` 文件创建，内容包含 `ts`、`request_messages`（完整 messages 数组）、`response_message`（单条助手响应）结构化字段。
- CLI：check 对合法/非法配置分别 exit 0/1，成功输出两个默认字段；list 表格输出 `model_name / model_id / enabled / type / base_url` 且不含 secret；start 对非法配置拒绝启动。
- 托盘：启动后创建单个托盘图标；重复启动不创建第二个服务；右键菜单按家族分组展示所有 enabled profile 并 radio 勾选当前默认;点击某项后 `cfgs/app.json` 对应字段被以缩进 2 的 JSON 写回,`running.runtime.app` 内存字段同步更新,下一次请求即生效;写回失败通过系统通知反馈且不修改内存;右键“退出”释放端口;配置错误和端口占用触发系统通知;调试模式不改变代理行为;打包 exe 从同级 `cfgs/`、`logs/` 读写,开发态沿用 `process.cwd()`。
- 配置管理窗口：点击“配置管理…”打开单个窗口且重复点击只聚焦不新建；表单保存前经 `validateConfig` 校验，失败不写盘且给出字段路径；校验通过后 `cfgs/app.json` 与 `cfgs/model_api.json` 以缩进 2 JSON 写回，`running.runtime` 同步更新并 rebuildTrayMenu；窗口不明文展示 `local_key` / `api_key`，日志中也无明文。
